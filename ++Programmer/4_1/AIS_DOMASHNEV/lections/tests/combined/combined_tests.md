### 1. На сервере serv01 хранится зона DNS companyA.local. Фрагмент содержимого этой зоны приведен ниже

 ┌──┬──────────────────────────────────────────────────────────┐
 │1 │@       IN  SOA serv02.companyA.local.hostmaster (        │
 │2 │        3       ; serial number                           │
 │3 │        900     ; refresh                                 │
 │4 │        600     ; retry                                   │
 │5 │        86400   ; expire                                  │
 │6 │        3600    ; default TTL                             │
 │7 │@           NS      serv01.companyE.local.                │
 │8 │@           NS      serv02.companyE.local.                │
 │9 │                                                          │
 │10│serv01      A       192.168.99.1                          │
 │11│serv02      A       192.168.99.2                          │
 │12│pc01        A       102.168.99.20                         │
 │13│)                                                         │
 └──┴──────────────────────────────────────────────────────────┘

Укажите тип этой зоны
a) Первичная
```
    b) Вторичная
```
c) Третичная
d) Зона-заглушки


___ Пояснение
Тип зоны определяется по тому, какой сервер является основным (primary) для неё, то есть где хранится оригинальная копия данных зоны.
В данном фрагменте зона DNS содержит SOA-запись, где указано, что основной сервер зоны — это serv02 (@ IN SOA serv02.companyA.local). Это означает, что сервер serv01 не является основным для этой зоны, а только хранит её копию.
Таким образом, сервер serv01 хранит вторичную копию зоны, полученную от основного сервера serv02.

### 2. DNS-сервер serv01 вышел из строя 15.05.2010 в 15:20:13. Последняя синхронизация зоны DNS companyF.local была проведена DNS-сервером serv02 15.05.2010 в 15:14:00. Фрагмент содержимого зоны companyF.local, расположенной на DNS-сервере serv02, приведен ниже.

┌──┬──────────────────────────────────────────────────┐
│1 │@       IN  SOA serv01.companyF.local.hostmaster (│
│2 │             2010    ; serial number              │
│3 │             900     ; refresh                    │
│4 │             600     ; retry                      │
│5 │             86400   ; expire                     │
│6 │             3600    ; default TTL                │
│7 │@       NS serv01.companyF.local.                 │
│8 │@       NS serv02.companyF.local.                 │
│9 │serv01  A  192.168.99.1                           │
│10│serv02  A  192.168.99.2                           │
│11│pc01    A  102.168.99.20                          │
│12│)                                                 │
└──┴──────────────────────────────────────────────────┘


До какого максиально возможного времени зона companyF.local будет работоспособна:
a) До 15:35:13 15.05.2010
b) До 15:20:13 16.05.2010
```
    c) До 15:29:00 16.05.2010
```
d) Время ничем не ограничено


___ Пояснение
В данном случае expire равен 86400 секунд, что соответствует 24 часам. Это означает, что вторичный сервер serv02 сможет обслуживать запросы до истечения этого времени с момента последней успешной синхронизации, которая произошла 15.05.2010 в 15:14:00.
Следовательно, зона companyF.local будет работоспособна до:
15:29:00 16.05.2010 (через 24 часа и 15 минут после последней синхронизации).

### 4. Какая модель репликации используется для поддержания в целостности содержимого зон DNS при хранении зон в файлах?
```
    a) Модель репликации с одним основным участником
```
b) Модель репликации с двумя участниками
c) Модель репликации с четырьмя участниками
d) Модель репликации со мнгогими равноправными участниками


___ Пояснение:
Для поддержания целостности содержимого зон DNS при хранении зон в файлах используется модель репликации, при которой существует один основной (primary) DNS-сервер, на котором хранится оригинальная копия зоны, и один или несколько вторичных (secondary) серверов, которые получают копии зоны через процессы репликации (трансфер зоны).
Это соответствует модели репликации с одним основным участником, где основной сервер вносит изменения в зону, а вторичные серверы синхронизируются с ним.

### 1. В приведенном фрагменте зоны DNS 99.168.192.in-addr.arpa вычеркинте все ошибочные ресурсные записи

 ┌──┬──────────────────────────────────────────────────────────┐
 │1 │@       IN  SOA serv01.companyE.local.hostmaster (        │
 │2 │        3       ; serial number                           │
 │3 │        900     ; refresh                                 │
 │4 │        600     ; retry                                   │
 │5 │        86400   ; expire                                  │
 │6 │        3600    ; default TTL                             │
 │7 │@       IN  NS      serv01.companyE.local.                │
 │8 │@       IN  NS      serv02.companyE.local.                │
 │9 │                                                          │
`│10│serv01      A       192.168.99.1                     <--- │
 │11│serv02      A       192.168.99.2                     <--- │
 │12│www         CNAME   serv01.companyE.local.                │`
 │13│1           PTR     serv01.companyE.local.                │
 │14│2           PTR     serv02.companyE.local.                │
 │15│)                                                         │
 └──┴──────────────────────────────────────────────────────────┘

___ Пояснение

В данном фрагменте представлена зона 99.168.192.in-addr.arpa, которая используется для обратного разрешения IP-адресов на имена (reverse DNS lookup). Основной тип записей в таких зонах — это PTR (Pointer), которые сопоставляют IP-адреса именам хостов.

Ошибочные ресурсные записи в этом фрагменте:

    A-записи для serv01 и serv02 (строки 10 и 11):
        В зонах обратного разрешения используются только записи PTR, а A-записи недопустимы. Они используются в прямых зонах для сопоставления имен хостов IP-адресам.
    CNAME-запись для www (строка 12):
        В зонах обратного разрешения CNAME-записи не используются, так как эта зона предназначена только для обратного преобразования IP-адресов в имена.

Корректные записи в этой зоне — только PTR-записи (строки 13 и 14), так как они используются для преобразования IP-адресов в имена хостов.

### 3. DNS-сервер serv01 был временно недоступен 12.05.2010 в период с 11:40:09 по 13:11:15. Непосредственно перед этим в 11:38:00 администратор добавил новую запись A для хоста pc02. Фрагмент содержимого зоны companyH.local, расположенной на DNS-сервере serv02, приведен ниже.

┌──┬──────────────────────────────────────────────────┐
│1 │@       IN  SOA serv01.companyH.local.hostmaster (│
│2 │             2010    ; serial number              │
│3 │             900     ; refresh                    │
│4 │             600     ; retry                      │
│5 │             86400   ; expire                     │
│6 │             3600    ; default TTL                │
│7 │@       NS serv01.companyH.local.                 │
│8 │@       NS serv02.companyH.local.                 │
│9 │serv01  A  192.168.99.1                           │
│10│serv02  A  192.168.99.2                           │
│11│pc01    A  102.168.99.20                          │
│12│)                                                 │
└──┴──────────────────────────────────────────────────┘


Как скоро сведения о хосте pc02 появятся на DNS-сервере serv02, если известно, что
последняя удачная сессия синхронизации была проведена 12.05.2010 в 11:29:16
a)  В 13:11:16  12.05.2010
```
    b)  В 13:26:16  12.05.2010
```
c)  В 13:14:16  12.05.2010
d)  Никогда не появятся


___ Пояснение
Последняя удачная сессия синхронизации была в 11:29:16. Следующий запрос на синхронизацию должен был произойти через 15 минут, т.е. в 11:44:16. Однако на момент синхронизации DNS-сервер serv01 был недоступен (с 11:40:09 по 13:11:15).
Таким образом, первый успешный запрос на синхронизацию произойдёт через retry-интервал после восстановления доступности сервера. serv01 становится доступен в 13:11:16, и следующий запрос будет через 600 секунд, то есть в 13:21:16.

Следовательно, правильный ответ:

b) В 13:26:16 12.05.2010.
